# MAX API и загрузка вложений

## Цель

Скачивать вложения из MAX-чата по выбранной дате, раскладывать платежные поручения отдельно от остальных файлов и сохранять исходные данные сообщений для будущего архива счетов.

## Текущий статус

- Токен читается из `.env`, в код и документацию не записывается.
- `GET /messages` работает с `Authorization` в заголовке.
- MAX фактически принимает `from/to` в миллисекундах.
- Рабочий диапазон дня: `from = конец дня МСК`, `to = начало дня МСК`.
- Вложения в живом JSON приходят как `filename` на уровне attachment и `payload.url` внутри payload.
- Программа умеет скачивать разные типы вложений, а не только PDF.

## Важные решения

- Дата MAX считается по МСК.
- Сообщения упорядочиваются по `body.seq`; `GET /messages` возвращает последние
  сообщения первыми.
- `body.text` и `body.attachments` принадлежат текущему сообщению. Объект
  `link.message` является отдельным ответным или пересланным сообщением и не
  должен рекурсивно смешиваться с внешним `body`.
- Точная техническая связь между сообщениями существует только для
  `link.type = reply` или `forward`. Простая близость по времени связь не
  доказывает.
- `body.mid` идентифицирует сообщение, а токен/ID вложения идентифицирует файл;
  эти идентификаторы нельзя взаимозаменять.
- ПП PDF идут в `_ПП/YYYY.MM.DD`.
- Остальные файлы идут в `_ПП/_прочее_MAX/YYYY.MM.DD`.
- Повторные скачивания контролируются журналом `.max_downloaded.json`.
- Документы с одинаковыми именами не считаются дублями без совпадения
  идентификатора или хеша содержимого.
- Для режима `ИС` позже добавляется отдельный chat id.

## Проверки

- Проверить живой `GET /messages` на текущем chat id.
- Проверить скачивание реального файла во временную папку.
- Проверить, что платежные поручения распознаются по содержимому PDF.
- Проверить, что счета, сканы и офисные файлы не попадают в `_ПП`.
- Проверить раздельное извлечение вложений из `body` и `link.message`.
- Проверить `reply`, `forward` и два разных файла с одинаковым именем.
