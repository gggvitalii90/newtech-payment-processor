## Что уже работает

- Чтение чатов MAX по потокам ПСК и ИС.
- Архив счетов в Google Sheets: лист `Архив счетов`.
- Скачивание счетов, распознавание PDF/изображений/документов и загрузка на Google Drive.
- Сопоставление счета и подписи из чата:
  - `reply` на конкретный счет считается самым надежным совпадением;
  - обычная подпись относится к предшествующему блоку неподписанных счетов того же сотрудника;
  - повторно отправленный одинаковый файл должен отсеиваться по содержимому/хэшу и не плодить дубли.
- Архив ПП: лист `Архив ПП`, ссылки на ПП на Google Drive, структура по годам/месяцам/датам.
- Итоговые листы: `Итоговая` и `Итоговая ИС`.
- Наличка из чата попадает в итоговую таблицу и синхронизируется в FinTablo.
- FinTablo используется как источник банковских транзакций и как место, куда дозаполняются статьи, направления, сделки и этапы.
- Ежедневный сценарий пишет дневные отчеты в Telegram со ссылкой на Google таблицу.
- Google Drive lifecycle: оплаченные счета переносятся в папку месяца по дате оплаты, неоплаченные остаются в корне объекта.

## Текущее состояние

Основной сценарий уже задеплоен и запускается на VPS. Ключевой файл сценария: `scripts/run_daily_update.py`.

Сейчас ежедневный запуск делает по каждому дню отдельно:

1. Обновляет архив счетов для ПСК и ИС.
2. Обновляет архив ПП и итоговые строки для ПСК и ИС.
3. Синхронизирует FinTablo по сгенерированной итоговой таблице.
4. Накладывает ручную итоговую таблицу Виталия как доверенный слой классификации.
5. Переносит оплаченные счета в правильные папки Google Drive.
6. Отправляет Telegram-отчет.

## Известные хвосты

- `unmatched_invoice` в отчете означает: ПП/операция есть, но счет не найден уверенно. Такие строки все равно должны попадать в итоговую, даже если часть классификации пустая.
- `FinTablo без строки в Итоговой` часто означает банковскую операцию, которой нет в чате: комиссии, обслуживание счета, приходы, зарплатные реестры, налоги или операции, которые бухгалтер не прислал в MAX.
- `FinTablo manual без совпадения` сейчас трактуется как очередь к сверке, а не как авария, если нет технических ошибок обновления.
- Часть правил распознавания еще дообучается по ручной таблице: назначения, статьи бюджета, проекты и ответственные.
- При появлении нового объекта в справочнике надо создать одноименную папку на Google Drive и вести ее по тем же правилам.

## Проверка перед релизом

Минимальная проверка после правок:

```powershell
python -m pytest tests/test_daily_update.py tests/test_telegram_notify.py tests/test_fintablo_sync_daily.py tests/test_fintablo_sync_from_manual_final.py
python -m compileall payment_processor scripts
```

Боевой/серверный сценарий:

```bash
cd /opt/newtech-payment-processor
START_DATE=2026-07-10 END_DATE=2026-07-11 ./deploy/run_daily.sh
```

Без записи и без Telegram:

```bash
cd /opt/newtech-payment-processor
DRY_RUN=1 START_DATE=2026-07-10 END_DATE=2026-07-11 ./deploy/run_daily.sh
```
